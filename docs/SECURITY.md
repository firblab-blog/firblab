# Security Architecture

This document is the authoritative security reference for the firblab homelab infrastructure. It covers the Vault HA cluster, secrets management, host hardening, certificate lifecycle, network isolation, Terraform secret integration, and the verification checklist used to validate the security posture of the entire environment.

For network topology and firewall rule details, see [NETWORK.md](NETWORK.md). For Vault operational procedures (unseal, backup, restore, rotation), see [VAULT-OPERATIONS.md](VAULT-OPERATIONS.md). For the overall system design, see [ARCHITECTURE.md](ARCHITECTURE.md).

## 1. Vault HA Cluster (3-Node Raft)

### 1.1 Cluster Topology

The production Vault cluster runs as a three-node Raft cluster spread across three physically distinct machines. This provides consensus-based high availability with automatic leader election and data replication.

| Node | Machine | OS | Role | Address |
|---|---|---|---|---|
| vault-1 | Mac Mini M4 | macOS 15+ (native LaunchDaemon) | Voter (initial leader) | 10.0.10.10:8200 |
| vault-2 | lab-02 (Proxmox VM, ID 2001) | Rocky Linux 9 AMD64 | Voter | 10.0.50.2:8200 |
| vault-3 | RPi5 CM5 | Ubuntu 24.04 ARM64 (bare metal) | Voter | 10.0.10.13:8200 |

**Quorum requirement:** 2 of 3 nodes must be online and reachable for the cluster to remain operational. The cluster tolerates the loss of any single node without service interruption. If two nodes are lost simultaneously, the cluster seals itself and becomes read-only until quorum is restored.

The nodes span two VLANs (Management VLAN 10 and Security VLAN 50) and two CPU architectures (ARM64 and AMD64). This diversity is intentional: a firmware bug, hypervisor vulnerability, or network segment failure is unlikely to affect all three nodes at the same time.

### 1.2 Mac Mini Setup

The Mac Mini M4 serves as the always-on anchor for the Vault cluster. Vault runs natively on macOS as a LaunchDaemon, eliminating the overhead and complexity of a hypervisor layer.

Key configuration details:

- **Networking:** The Mac Mini has a static IP address (10.0.10.10) on the Management VLAN (10), assigned by the gw-01 DHCP server via MAC reservation.
- **Disk encryption:** FileVault is enabled on the macOS host. This means the Vault data directory (`/opt/vault/data`) is encrypted at rest by the host operating system. Even if the Mac Mini's storage is physically removed, the Vault data is protected by FileVault's XTS-AES-128 encryption.
- **Host firewall:** The macOS `pf` firewall is configured to allow traffic only to the Vault-related ports (8200 for the production Vault API, 8210 for the unseal Vault, and 8201 for Raft cluster communication). All other inbound connections are dropped.
- **Process security:** Vault runs as a dedicated `vault` user with a LaunchDaemon managed by launchd. The process has `mlock` capabilities to prevent Vault memory from being swapped to disk.

### 1.3 Raft Storage Configuration

All three nodes use the integrated Raft storage backend. Each node's configuration includes `retry_join` stanzas pointing to all three cluster members, allowing any node to find and rejoin the cluster after a restart without manual intervention.

```hcl
storage "raft" {
  path    = "/opt/vault/data"
  node_id = "vault-1"   # vault-2, vault-3 on other nodes

  retry_join { leader_api_addr = "https://10.0.10.10:8200" }
  retry_join { leader_api_addr = "https://10.0.50.2:8200" }
  retry_join { leader_api_addr = "https://10.0.10.13:8200" }
}

listener "tcp" {
  address       = "0.0.0.0:8200"
  tls_cert_file = "/etc/vault.d/tls/vault-cert.pem"
  tls_key_file  = "/etc/vault.d/tls/vault-key.pem"
}
```

TLS is mandatory on the listener. Plaintext HTTP is never used, even during bootstrap. The initial TLS certificates are generated by Ansible during deployment and later rotated by the Vault PKI engine once the cluster is operational.

### 1.4 Auto-Unseal Architecture

Vault's Raft storage is encrypted at rest. After a restart, each node is in a "sealed" state and cannot serve requests until it is unsealed. Rather than requiring manual Shamir key entry on every reboot, the production cluster uses Transit Auto-Unseal backed by a dedicated lightweight Vault instance.

**Unseal Vault:**

A separate, minimal Vault process runs on the same Mac Mini as vault-1, listening on port 8210. This unseal Vault serves a single purpose: it hosts a Transit secrets engine with a key named `autounseal` that the production cluster uses to decrypt its master key on startup.

The unseal Vault uses Shamir 1/1 — a single key share with a threshold of one. This is acceptable because the unseal Vault protects only the transit encryption key, not any application secrets. Its threat model is limited to: "after a full power outage, one manual step is required to bring the cluster back."

The production nodes reference the unseal Vault in their seal stanza:

```hcl
seal "transit" {
  address    = "https://127.0.0.1:8210"   # Local on vault-1; remote on vault-2/vault-3
  token      = "<unseal-vault-token>"
  key_name   = "autounseal"
  mount_path = "transit/"
}
```

For vault-2 and vault-3, the `address` field points to the Mac Mini's network address (10.0.10.10:8210) rather than localhost.

**Recovery procedure after a full power outage:**

1. The Mac Mini boots automatically (configured in macOS Energy Saver to restart after power loss).
2. Vault starts automatically via the LaunchDaemon (launchd).
3. An operator manually unseals the unseal Vault: `vault operator unseal -address=https://10.0.10.10:8210`.
4. Once the unseal Vault is running and unsealed, all three production Vault nodes auto-unseal within seconds via the Transit seal mechanism.

**Unseal key storage (3 copies, 3 locations):**

| Copy | Location | Format |
|---|---|---|
| Primary | Password manager (1Password / Bitwarden) | Encrypted digital entry |
| Secondary | Physical safe | Printed QR code on paper |
| Tertiary | RPi5 CM5 | age-encrypted file on disk |

No single point of failure can permanently lock out the unseal key. The password manager is accessible from any device, the printed QR survives digital failures, and the RPi-hosted copy survives if the physical safe is inaccessible.

## 2. Secrets Hierarchy in Vault

All secrets consumed by infrastructure automation, application deployments, and operational tooling are stored in HashiCorp Vault under a structured path hierarchy. No secrets are stored in plaintext anywhere in the Git repository, in environment variables on CI runners, or in configuration files on disk.

### 2.1 Path Layout

The KV v2 secrets engine uses a taxonomy that separates secrets by their role in the infrastructure. This makes policy scoping natural and keeps `vault kv list` output clean at every level.

```
vault/
├── secret/                              # KV v2 secrets engine
│   ├── infra/                           # Infrastructure device credentials
│   │   ├── proxmox/
│   │   │   ├── lab-01               # { url, token_id, token_secret }
│   │   │   ├── lab-02               # { url, token_id, token_secret }
│   │   │   ├── lab-03               # { url, token_id, token_secret }
│   │   │   └── lab-04               # { url, token_id, token_secret }
│   │   ├── unifi/                       # { url, username, password, api_key }
│   │   ├── hetzner/                     # { token, s3_access_key, s3_secret_key, s3_endpoint }
│   │   └── cloudflare/                  # { token, zone_id }
│   ├── compute/                         # Per-host secrets (SSH keys, admin passwords)
│   │   ├── lab-01                   # { ssh_private_key, ssh_public_key }
│   │   ├── lab-02                   # { ssh_private_key, ssh_public_key }
│   │   └── ...                          # (per-host entries for all managed hosts)
│   ├── services/                        # Application-level secrets
│   │   ├── gitlab/
│   │   │   ├── admin                    # { root_password, runner_token }
│   │   │   └── approle                  # { role_id, secret_id }
│   │   ├── ghost                        # { db_password, admin_password, mail_password }
│   │   ├── foundryvtt                   # { admin_password, license_key }
│   │   ├── mealie                       # { api_token, db_password }
│   │   ├── sonarqube                    # { db_username, db_password }
│   │   ├── netbox                       # { db_password, secret_key, superuser_password, api_token }
│   │   ├── scanopy                      # { db_password, admin_email, admin_password, daemon_network_id, daemon_api_key }
│   │   ├── nut                          # { monitor_password, admin_password }
│   │   └── wireguard/peer1             # { private_key, preshared_key }
│   ├── k8s/                             # Kubernetes-specific secrets (External Secrets Operator)
│   │   ├── grafana                      # { username, password }
│   │   ├── longhorn                     # { username, password } (basic-auth for UI)
│   │   ├── longhorn-s3                  # { s3_access_key, s3_secret_key, s3_endpoint } (S3 backup creds)
│   │   └── headlamp                     # { token } (ServiceAccount token for K8s dashboard)
│   ├── backup/                          # Backup & DR credentials
│   │   ├── age-key                      # { public_key } (age encryption for all backups)
│   │   └── vault                        # { token } (Vault backup token for raft snapshots)
├── pki/
│   ├── root-ca/                         # Root CA (offline, long TTL)
│   └── intermediate-ca/                 # Intermediate CA (issues certs, short TTL)
└── transit/
    └── autounseal/                      # Auto-unseal key for production cluster
```

**Design principles:**

| Principle | Rationale |
|---|---|
| `infra/` vs `compute/` vs `services/` | Clear separation: devices that host things, the machines themselves, apps running on them |
| `k8s/` is separate from `services/` | k8s secrets are synced by External Secrets Operator with different access patterns |
| Node name as map key, not IP | `infra/proxmox/lab-02` not `infra/proxmox/10.0.10.2` — IPs change, names don't |
| Max 2 levels deep within categories | Keeps `vault kv list` output usable at every level |
| `backup/` is its own concern | DR credentials must be reachable even when the thing they protect is down |

### 2.2 Secrets Engine Details

**`secret/` (KV v2):** The key-value version 2 engine with versioning enabled. Every write creates a new version, and previous versions are retained (default: 10 versions). This provides an audit trail and the ability to roll back a bad secret rotation without downtime.

- **Infrastructure credentials** (`infra/*`): API tokens and credentials for infrastructure devices — Proxmox nodes (all 4), UniFi controller, Hetzner (API + S3), Cloudflare. Consumed by Terraform at plan/apply time via the Vault provider (`data "vault_kv_secret_v2"`) and by Ansible during configuration management. These are long-lived API tokens with the minimum permissions required for automation.
- **Compute secrets** (`compute/*`): Per-host SSH keys and admin passwords. ED25519 private keys generated per host by Terraform modules. Stored in Vault immediately after generation and never written to the local filesystem outside of ephemeral use.
- **Service credentials** (`services/*`): Database passwords, admin passwords, API tokens, and license keys for each deployed application. Consumed by Ansible roles during initial deployment and by the External Secrets Operator for Kubernetes workloads.
- **Kubernetes secrets** (`k8s/*`): Secrets consumed by k8s platform components (ArgoCD, cert-manager, Longhorn, monitoring). Synced into the cluster by the External Secrets Operator. Separated from `services/` because they have different access patterns (service accounts, not tokens).
- **Backup credentials** (`backup/*`): The age public key for encrypting all backup data, and a dedicated Vault token for Raft snapshot operations. S3 credentials for off-site uploads are stored at `infra/hetzner` (shared with other Hetzner operations).

**`pki/` (PKI Secrets Engine):** Two mounts — a root CA and an intermediate CA. The root CA has a long TTL (10 years) and is used only to sign the intermediate CA certificate. The intermediate CA has a shorter TTL (1 year) and issues all operational certificates. See Section 4 (Certificate Management) for full details.

**`transit/` (Transit Secrets Engine):** Hosts the `autounseal` key used by the production cluster's seal stanza. This engine runs on the unseal Vault instance (port 8210), not on the production cluster. The key is AES-256-GCM and is never exported.

### 2.3 Access Policies

Vault policies follow the principle of least privilege. Each consumer of secrets receives a policy that grants access only to the specific paths it needs.

Policies are managed declaratively by Terraform (Layer 02-vault-config). Each consumer gets access to exactly the taxonomy paths it needs:

| Policy | Paths | Use Case |
|---|---|---|
| `admin` | `secret/*` (full CRUD) | Day-to-day admin via CLI |
| `terraform` | `secret/data/infra/*`, `secret/data/compute/*` (read) | Terraform provisioning |
| `gitlab-ci` | `secret/data/infra/*`, `secret/data/services/*`, `secret/data/k8s/*` (read) | CI/CD pipelines |
| `k8s-external-secrets` | `secret/data/k8s/*`, `secret/data/services/*` (read) | External Secrets Operator |
| `backup` | `secret/data/backup/*` (read), `sys/storage/raft/snapshot` (read) | Automated backup scripts |
| `cert-manager` | `pki_int/issue/firblab` (create, update) | Kubernetes cert issuance |

No consumer has access to `secret/data/backup/` except the backup automation identity. No consumer has access to the Transit engine on the unseal Vault. The Vault root token is revoked after initial setup and is only regenerated (via `vault operator generate-root`) for emergency break-glass scenarios.

### 2.4 Audit Logging

Vault audit logging is enabled on all nodes. Every request and response (with sensitive fields HMAC-hashed) is written to a file audit device at `/var/log/vault/audit.log`.

Audit logging serves two purposes: forensic investigation of any unauthorized access attempt, and compliance evidence that secrets are being accessed only by authorized identities. When Wazuh is deployed, these logs will be forwarded to the Wazuh Manager for centralized analysis and alerting.

## 3. Hardening Baseline

Every host in the infrastructure — Proxmox nodes, standalone VMs, LXC containers, the Mac Mini VM, the RPi, and the Hetzner VPS — receives a consistent security hardening baseline. This baseline is applied by Ansible and is enforced on every configuration management run, not just at initial provisioning.

### 3.1 Linux Hardening (All Hosts)

Applied via `ansible/playbooks/harden.yml`, the hardening playbook applies the following controls in order:

**DevSec OS Hardening.** The `devsec.hardening.os_hardening` role from the DevSec Hardening Framework applies a comprehensive set of operating system security configurations. This includes: restricting core dumps, setting appropriate permissions on sensitive files (`/etc/shadow`, `/etc/gshadow`), disabling unused filesystems (cramfs, freevfat, squashfs, udf), configuring sysctl parameters for network stack hardening (SYN cookies, reverse path filtering, ICMP redirect rejection), and removing unnecessary packages and services.

**DevSec SSH Hardening.** The `devsec.hardening.ssh_hardening` role locks down the SSH daemon configuration. Password authentication is disabled. Root login is disabled. Only ED25519 and RSA (4096-bit minimum) host keys are permitted. Idle sessions are terminated after a configurable timeout. SSH agent forwarding and X11 forwarding are disabled. The `AllowUsers` directive restricts SSH access to the designated admin user.

**CIS Ubuntu 24.04 Level 1.** The CIS (Center for Internet Security) Ubuntu 24.04 Level 1 benchmark is applied. This covers filesystem configuration (separate partitions for `/tmp`, `/var`, `/var/log`), boot loader hardening (GRUB password), process hardening, network configuration (IPv6 disabled where not needed), logging configuration (rsyslog and journald retention), access and authentication (PAM configuration, password quality, account lockout), and system maintenance (file permission auditing).

**fail2ban.** The fail2ban intrusion prevention framework is installed and configured on every host. It monitors SSH authentication logs and bans IP addresses that exceed the failure threshold (default: 5 failures in 600 seconds, ban duration 3600 seconds). On internet-facing hosts (Hetzner, DMZ), additional jails are configured for HTTP authentication endpoints.

**Wazuh Agent.** (Not currently deployed — Wazuh Manager is not yet available; see Section 3.3.) When deployed, every host would run a Wazuh agent that reports to the centralized Wazuh Manager. The agent would provide: real-time log analysis and correlation, file integrity monitoring (FIM), rootkit detection, system inventory collection, and vulnerability scanning.

**CrowdSec Agent.** Internet-facing hosts (the Hetzner VPS and any services in the DMZ VLAN) run the CrowdSec security engine. CrowdSec parses logs in real time, detects attack patterns (brute force, credential stuffing, path traversal, scanner activity), and applies remediation (IP bans via the CrowdSec bouncer). It also participates in the CrowdSec community blocklist, receiving threat intelligence from the global network of CrowdSec installations.

**Automatic Security Updates.** The `unattended-upgrades` package is configured on all Debian/Ubuntu hosts to automatically install security updates from the `${distro_codename}-security` repository. Updates are applied daily. The system is configured to automatically reboot if required (at a scheduled maintenance window) and to email the admin if a reboot is pending.

**auditd.** The Linux Audit Framework is installed with a rule set based on the CIS benchmark recommendations. Key audited events include: changes to authentication configuration (`/etc/pam.d/`, `/etc/passwd`, `/etc/shadow`), use of privileged commands (`sudo`, `su`), changes to audit configuration itself, kernel module loading, file deletion events, and modifications to system time. Audit logs are forwarded to the Wazuh Manager.

**AIDE File Integrity Monitoring.** AIDE (Advanced Intrusion Detection Environment) is configured to maintain a database of file checksums for critical system files and directories. A daily cron job compares the current filesystem state against the baseline database and reports any unauthorized changes. AIDE operates independently of Wazuh FIM as a defense-in-depth measure — if one system is compromised or misconfigured, the other still provides coverage.

### 3.2 macOS-Specific Hardening

The Mac Mini M4 runs macOS as its host operating system. A dedicated playbook (`ansible/playbooks/harden-macos.yml`) applies macOS-specific security controls:

**FileVault Full-Disk Encryption.** FileVault is enabled to encrypt the entire startup volume with XTS-AES-128 encryption. The recovery key is stored in the password manager alongside the Vault unseal key. This protects the Vault data directory (`/opt/vault/data`) against physical theft or unauthorized disk access.

**macOS pf Firewall.** The BSD packet filter (`pf`) is configured and enabled. The ruleset allows inbound traffic only on ports 8200 (production Vault API), 8201 (Raft cluster communication), and 8210 (unseal Vault API). SSH (port 22) is allowed only from the Management VLAN subnet (10.0.10.0/24). All other inbound connections are blocked by default.

**Automatic Updates.** macOS Software Update is configured to automatically download and install system updates, including security patches and XProtect definitions. This is enforced via the `com.apple.SoftwareUpdate` preference domain.

**SSH Access Restriction.** Remote Login (SSH) is enabled but restricted to the designated admin user account. The `AllowUsers` directive in `sshd_config` ensures that only the admin account can authenticate, and password authentication is disabled (key-only).

**Screen Lock.** The screen saver activates after 5 minutes of idle time, and a password is required immediately upon activation. This prevents unauthorized physical access to the macOS session while the machine is unattended.

### 3.3 Wazuh SIEM/EDR

**Status:** Not deployed. Wazuh Manager was planned for lab-02 (Security VLAN 50) but removed due to RAM constraints (16 GB fully allocated to vault-2 + GitLab + Runner). The `wazuh-agent` role exists but is globally disabled (`wazuh_enabled: false` in Ansible). Re-evaluation pending — would require a host with sufficient RAM (8+ GB free).

**Planned capabilities (when deployed):**

- Log analysis and correlation across all hosts
- File integrity monitoring with real-time alerts on unauthorized changes
- Rootkit detection scans (scheduled daily)
- Vulnerability detection against the CVE database
- Security Configuration Assessment (SCA) against CIS benchmarks
- Active response for automated threat mitigation
- Vault audit log ingestion and alert rules for suspicious access patterns

## 4. Certificate Management

TLS is required on every service in the infrastructure. No plaintext HTTP is permitted, even on internal networks. The certificate strategy uses different issuance methods depending on whether a service is public-facing or internal, and whether it runs in Kubernetes or as a standalone deployment.

### 4.1 Certificate Issuance Matrix

| Scope | Method | Tool | TTL |
|---|---|---|---|
| Public-facing (Hetzner) | ACME / Let's Encrypt | Traefik auto-provisioning | 90 days (auto-renewed) |
| Public-facing (homelab via Hetzner) | ACME / Let's Encrypt | Traefik on Hetzner | 90 days (auto-renewed) |
| Internal RKE2 ingress | Vault PKI intermediate CA | cert-manager with Vault issuer | 30 days (auto-renewed) |
| Service-to-service (internal) | Vault PKI (short-lived) | Vault Agent / CSI driver | 24 hours (auto-renewed) |
| Vault cluster TLS | Vault PKI or manually generated (bootstrap) | Ansible | 1 year (manually rotated) |
| Proxmox API | Vault PKI intermediate CA | Ansible | 1 year (Ansible-rotated) |

### 4.2 Public Certificate Lifecycle

All public-facing services are fronted by Traefik running on the Hetzner VPS. Traefik is configured with the ACME protocol to automatically request, validate (DNS-01 challenge via Cloudflare API), and renew Let's Encrypt certificates for all configured domains. No manual certificate management is required for public services.

Homelab services that are exposed to the internet (Ghost, FoundryVTT, etc.) are accessed through the WireGuard site-to-site tunnel from the Hetzner VPS. Traefik on Hetzner terminates TLS with a Let's Encrypt certificate and proxies the request over the encrypted WireGuard tunnel to the internal service. The internal service also runs TLS (using a Vault-issued certificate), providing encryption in transit end-to-end.

### 4.3 Internal PKI Hierarchy

The Vault PKI secrets engine provides a private certificate authority for all internal services.

**Root CA** (`pki/root-ca/`):
- Self-signed, generated during Vault cluster bootstrap
- TTL: 10 years
- Used only to sign the intermediate CA certificate
- After signing the intermediate, the root CA role is effectively offline — no new certificates are issued directly from the root

**Intermediate CA** (`pki/intermediate-ca/`):
- Signed by the root CA
- TTL: 1 year (renewed annually by re-signing from the root)
- Issues all operational certificates for internal services
- Configured with roles that restrict the allowed domains (e.g., `*.example-lab.local`, `*.svc.cluster.local`)

### 4.4 Kubernetes Certificate Integration

Inside the RKE2 cluster, cert-manager is deployed with a Vault Issuer that authenticates to Vault via Kubernetes Service Account Token (JWT auth). When a Kubernetes Ingress or Certificate resource is created, cert-manager requests a certificate from the Vault intermediate CA, stores it as a Kubernetes Secret, and automatically renews it before expiration.

For pod-to-pod communication, the External Secrets Operator or Vault Agent Injector can provision short-lived (24-hour) certificates directly into pod filesystems, ensuring that even if a certificate is compromised, it expires quickly.

### 4.5 Bootstrap Certificate Problem

During initial deployment, the Vault cluster itself needs TLS certificates before the Vault PKI engine exists. This chicken-and-egg problem is resolved by Ansible: the `vault-deploy.yml` playbook generates self-signed or locally-CA-signed certificates for the initial Vault cluster TLS. Once the cluster is operational and the PKI engine is configured, these bootstrap certificates are replaced with Vault-issued certificates in a subsequent Ansible run.

## 5. VLAN Isolation

Network segmentation is the first line of defense. All traffic between security zones passes through the gw-01 firewall (UDM Pro), which enforces inter-VLAN rules managed by Terraform (Layer 00).

For the complete VLAN layout, subnet assignments, DHCP ranges, and the full firewall rule matrix, see [NETWORK.md](NETWORK.md).

### 5.1 Security-Relevant Isolation Rules

The following isolation principles are enforced at the network layer:

**Security VLAN (50) is isolated.** The Security VLAN hosts vault-2 (Vault standby node). GitLab is on Management VLAN 10; Wazuh is not deployed. The VLAN permits no inbound connections from the Default/LAN, DMZ, or Storage VLANs. The only allowed inbound traffic comes from the Management VLAN (for administrative access) and the Services VLAN (for Vault API on port 8200).

**DMZ VLAN (30) cannot reach Management or Storage.** The DMZ hosts the WireGuard endpoint and any internet-facing services on the homelab network. It can route to specific IP addresses on the Services VLAN (for reverse proxy backends) but is blocked from reaching the Management VLAN (where Proxmox hosts and the Mac Mini live) and the Storage VLAN (where NFS and Longhorn replication traffic flows).

**Storage VLAN (40) accepts only; it does not initiate.** The Storage VLAN carries NFS, iSCSI, and Longhorn replication traffic. Hosts on this VLAN do not initiate outbound connections. Egress filtering is enforced at the host level (`ufw_restrict_egress=true` in `group_vars/storage_vlan.yml`). Only the Services VLAN (for NFS/iSCSI from RKE2 workers) and the Management VLAN (for administrative access) can reach it.

**Default LAN (1) is untrusted.** The Default/LAN VLAN (home devices, FIOS router) is treated as an untrusted network. It has no access to any lab VLAN. Lab services that need to be accessed from personal devices are exposed through the Hetzner reverse proxy or via WireGuard VPN.

### 5.2 Verification

VLAN isolation is validated by running port scans from a host on each VLAN and confirming that only the expected ports are reachable on hosts in other VLANs. This scan is part of the security verification checklist (Section 7) and should be repeated after any change to the firewall rules in Layer 00.

## 6. Terraform Integration with Vault

All Terraform layers that manage infrastructure after the Vault cluster is operational read their credentials from Vault rather than from local `.tfvars` files or environment variables. This eliminates plaintext secrets from the CI/CD pipeline and from operator workstations.

### 6.1 Authentication

During bootstrap (before GitLab CI exists), Terraform authenticates to Vault using a token passed via the `VAULT_TOKEN` environment variable. This token is scoped to a policy that allows only the specific `secret/data/*` paths needed for the layer being applied.

After GitLab CI is operational, Terraform authenticates via Vault's AppRole auth method. The GitLab Runner has a Role ID and Secret ID (the Secret ID is a wrapped token with a short TTL) that it uses to obtain a Vault token at the start of each pipeline run. This token is scoped to the minimum permissions required for the layer.

### 6.2 Secret Retrieval Pattern

Every Terraform layer that needs infrastructure credentials follows this pattern:

```hcl
provider "vault" {
  address      = var.vault_addr
  token        = var.vault_token
  ca_cert_file = pathexpand(var.vault_ca_cert)
}

# Read Proxmox API credentials from Vault (KV v2)
data "vault_kv_secret_v2" "proxmox" {
  mount = "secret"
  name  = "infra/proxmox/${var.proxmox_node}"
}

# Use the retrieved credentials to configure the Proxmox provider
provider "proxmox" {
  endpoint  = data.vault_kv_secret_v2.proxmox.data["url"]
  api_token = "${data.vault_kv_secret_v2.proxmox.data["token_id"]}=${data.vault_kv_secret_v2.proxmox.data["token_secret"]}"
}
```

The same pattern applies to Hetzner, Cloudflare, and UniFi providers:

```hcl
data "vault_kv_secret_v2" "hetzner_api" {
  mount = "secret"
  name  = "infra/hetzner/api"
}

provider "hcloud" {
  token = data.vault_kv_secret_v2.hetzner_api.data["token"]
}

data "vault_kv_secret_v2" "cloudflare_api" {
  mount = "secret"
  name  = "infra/cloudflare/api"
}

provider "cloudflare" {
  api_token = data.vault_kv_secret_v2.cloudflare_api.data["token"]
}
```

### 6.3 State File Security

Terraform state files may contain sensitive values (provider credentials appear in state after a plan/apply). State files are protected at two levels:

- **Layers 00-02 (local state):** State files are stored locally and excluded from Git by `.gitignore`. To back up state in Git, encrypt with SOPS/age first (e.g., `sops --encrypt terraform.tfstate > terraform.tfstate.sops.json`). The `.sops.yaml` configuration defines encryption rules for `*.tfstate` files. The `*.sops.*` pattern is allowed through `.gitignore`. The age private key is stored in Vault (`secret/backup/age-key`) and in the password manager.
- **Layers 03-08 (local state, pending S3 migration):** State files are currently stored locally on the operator workstation and excluded from Git by `.gitignore`. CI plan jobs run against empty state and are configured with `allow_failure: true` as a workaround. **Planned:** Migrate to Hetzner Object Storage (S3-compatible) with server-side encryption. Access will require S3 credentials from Vault. State locking will be provided by GitLab CI pipeline serialization (only one apply per layer at a time). See `docs/DEPLOYMENT.md` Phase 2 for migration steps.

## 7. Security Verification Checklist

This checklist is executed after every major infrastructure change and as a full sweep during Phase 9 of the bootstrap sequence. Every item must pass before the infrastructure is considered production-ready.

### 7.1 Static Analysis

**tfsec scan: 0 critical/high findings.** Run `tfsec` against all Terraform layers. Any critical or high-severity finding must be resolved or explicitly risk-accepted with a documented reason in a `tfsec:ignore` comment. Medium and low findings are reviewed and addressed on a best-effort basis.

**trivy config scan: 0 critical findings.** Run `trivy config` against all Terraform, Kubernetes, and Docker Compose configurations. Critical findings are blocking. High findings must be reviewed and either resolved or documented.

**ansible-lint passes on all playbooks.** Run `ansible-lint` against the entire `ansible/` directory. All playbooks and roles must pass without errors. Warnings are reviewed and addressed where practical.

### 7.2 Compliance Benchmarks

**CIS Ubuntu 24.04 Level 1 benchmark: pass on all hosts.** The Wazuh SCA (Security Configuration Assessment) module runs the CIS Ubuntu 24.04 Level 1 benchmark on every host. All checks must pass. Exceptions are documented in the Wazuh SCA policy override configuration with a justification (for example, disabling IPv6 on a host that does not use it is an acceptable deviation if the benchmark expects it enabled).

### 7.3 Runtime Validation

**Wazuh: all agents reporting, no critical alerts.** Every managed host must have a Wazuh agent connected to the Manager with a status of "Active." The file integrity monitoring baseline must be initialized. No critical-severity alerts should be unacknowledged.

**Vault: audit log enabled, all policies least-privilege, root token revoked.** Confirm that `vault audit list` shows an enabled audit device. Review all policies with `vault policy list` and `vault policy read <name>` to confirm they grant only the minimum required permissions. Confirm the root token has been revoked with `vault token lookup` (should return an error for the root token).

**SSH: key-only authentication on all hosts.** Verify by attempting `ssh -o PasswordAuthentication=yes -o PubkeyAuthentication=no <host>` from a management workstation. The connection must be rejected. This confirms that `PasswordAuthentication no` is effective in the sshd configuration.

**VLAN isolation: port scan confirms only expected ports open.** From a host on each VLAN, run `nmap` against hosts on other VLANs. Only the ports permitted by the inter-VLAN firewall rules (documented in NETWORK.md) should be reachable. Any unexpected open port indicates a firewall rule misconfiguration.

**TLS: no plaintext HTTP on any internal service.** Attempt to connect to every service on port 80 (HTTP). The connection should either be refused, redirected to HTTPS, or return a TLS handshake error. No service should serve content over unencrypted HTTP.

**Secrets: no plaintext secrets in Git.** Run `git log -p --all | grep -iE '(password|secret|token|api_key)' ` against the repository history. No matches should return actual credential values. `.tfvars.example` files with placeholder values like `CHANGE_ME` are expected and acceptable.

**Backup: Vault restore test on fresh node succeeds.** Provision a clean Ubuntu VM, install Vault, and restore the latest Raft snapshot with `vault operator raft snapshot restore`. Confirm the restored Vault contains the expected secrets and PKI configuration. This test validates both the backup integrity and the restore procedure.

**UniFi: no manual configuration changes outside Terraform state.** Run `terraform plan` on Layer 00 (network). The plan should show no changes. Any drift indicates a manual change was made via the UniFi UI, which must be imported into Terraform or reverted.

## 8. Incident Response

### 8.1 Compromised Secret

If a secret is suspected to be compromised:

1. Immediately rotate the affected credential in Vault (`vault kv put` with a new value).
2. Trigger redeployment of any services consuming that secret (Ansible re-run or Kubernetes rollout restart).
3. Review Vault audit logs to determine when and how the secret was accessed.
4. Review Wazuh alerts for correlated suspicious activity on the affected hosts.
5. If the compromise vector is identified, remediate it before considering the incident closed.

### 8.2 Vault Seal Event

If the production Vault cluster seals unexpectedly (all nodes sealed, not just a single node restart):

1. Check the unseal Vault status: `vault status -address=https://10.0.10.10:8210`.
2. If the unseal Vault is sealed, unseal it using the Shamir key from the password manager.
3. Once the unseal Vault is running, the production nodes should auto-unseal. Monitor with `vault status` on each node.
4. If auto-unseal fails, check network connectivity between nodes and review Vault server logs.
5. Investigate the root cause of the seal event (power outage, node crash, storage failure).

### 8.3 Node Compromise

If a host is suspected to be compromised:

1. Isolate the host by disabling its switch port or VLAN membership (via UniFi UI or emergency Terraform apply).
2. If the host is a Vault node, verify cluster health on the remaining nodes (`vault operator raft list-peers`). The cluster tolerates one node loss.
3. Capture forensic data: memory dump (if possible), disk image, Wazuh agent logs, auditd logs.
4. Revoke any Vault tokens or credentials that were present on the compromised host.
5. Rebuild the host from scratch using the Ansible playbooks (never attempt to "clean" a compromised system).
6. Re-enroll the rebuilt host in Wazuh and verify all hardening controls are applied.

## 9. Multi-Factor Authentication (MFA)

MFA should be enabled on every administrative interface to prevent credential-based compromise. The following services support MFA and should be configured during or immediately after initial deployment.

### 9.1 GitLab (Built-in TOTP)

GitLab CE has built-in TOTP-based two-factor authentication. Enable it for all admin accounts:

1. Log in to GitLab web UI as admin.
2. Navigate to **User Settings > Account > Two-Factor Authentication**.
3. Scan the QR code with an authenticator app (1Password, Authy, Google Authenticator).
4. Enter the verification code to confirm.
5. Save the recovery codes in the password manager.
6. **Enforce 2FA for all users:** Admin Area > Settings > General > Sign-in restrictions > "Require all users to set up two-factor authentication."

### 9.2 Vault (Userpass + TOTP Auth Method)

For interactive admin access to Vault, layer TOTP on top of the userpass auth method:

```bash
# Enable TOTP secrets engine
vault secrets enable totp

# Generate a TOTP key for the admin user
vault write totp/keys/admin \
  generate=true \
  issuer="firblab-vault" \
  account_name="admin"
# Scan the returned barcode_uri QR code with your authenticator app

# Validate with: vault read totp/code/admin
```

Machine-to-machine auth (AppRole, Kubernetes JWT) does not use MFA — those identities are authenticated by cryptographic credentials and scoped policies.

### 9.3 Proxmox Web UI

Proxmox VE supports TOTP natively:

1. Log in to Proxmox web UI.
2. Navigate to **Datacenter > Permissions > Two Factor**.
3. Add a TOTP factor for each admin user.
4. The next login will require both password and TOTP code.

### 9.4 WireGuard VPN

WireGuard itself uses public key cryptography and does not support MFA natively. Compensating controls:

- WireGuard private keys are treated as sensitive as passwords — stored in the password manager, never on disk unprotected.
- Peer configs use pre-shared keys (PSK) for post-quantum defense-in-depth.
- AllowedIPs restricts each peer to the minimum required subnets.
- Consider adding a WireGuard auth gateway (e.g., Firezone or wg-access-server) if more granular access control is needed in the future.

## 11. Immutable Backups

### 11.1 Hetzner Object Storage with Object Lock

Hetzner Object Storage supports S3-compatible object lock (WORM — Write Once Read Many). This prevents backup deletion or modification even if S3 credentials are compromised.

**Configuration:**

```bash
# Create bucket with object lock enabled (must be set at bucket creation)
aws s3api create-bucket \
  --bucket firblab-backups-immutable \
  --endpoint-url https://region1.your-objectstorage.com \
  --object-lock-enabled-for-bucket

# Set default retention policy (e.g., 30 days governance mode)
aws s3api put-object-lock-configuration \
  --bucket firblab-backups-immutable \
  --endpoint-url https://region1.your-objectstorage.com \
  --object-lock-configuration '{
    "ObjectLockEnabled": "Enabled",
    "Rule": {
      "DefaultRetention": {
        "Mode": "GOVERNANCE",
        "Days": 30
      }
    }
  }'
```

**Governance vs. Compliance mode:**
- **Governance:** Users with `s3:BypassGovernanceRetention` permission can delete objects before retention expires. Use this if you need the ability to clean up in emergencies.
- **Compliance:** No one can delete objects before retention expires, not even the account owner. Use this for ransomware-proof backups where you accept the tradeoff.

### 11.2 Separate Credentials

The credentials for the immutable backup bucket should be stored separately from the primary S3 credentials at `secret/infra/hetzner`. The backup automation identity should have write-only access (no delete permission on the immutable bucket). This is a future enhancement — current backups use standard S3 buckets with age encryption.
